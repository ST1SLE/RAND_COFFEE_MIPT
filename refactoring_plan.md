### Архитектура v2.0

Вместо одного процесса `python bot.py` у нас будет 3 контейнера в `docker-compose`:
1.  **db**: PostgreSQL с установленным расширением `pgvector`.
2.  **bot**: Отвечает только за интерфейс, сбор данных (Bio) и уведомления.
3.  **worker**: Отдельный сервис с ML-библиотеками (`torch`, `sentence-transformers`). Занимается тяжелыми вычислениями (генерация эмбеддингов, расчет матрицы сходства, мэтчинг).

---

### План Рефакторинга

#### Этап 1: Инфраструктура и База Данных
*Цель: Подготовить среду для работы с векторами и микросервисами.*

1.  **Подготовка Docker окружения**:
    *   Создать `Dockerfile` для `bot` (легковесный).
    *   Создать `Dockerfile.ml` для `worker` (с `torch` и `transformers` — образ будет тяжелее).
    *   Создать `docker-compose.yml`, связав сервисы и прокинув переменные окружения.
2.  **Миграция БД (pgvector)**:
    *   Написать SQL-скрипт миграции:
        *   Включить расширение `CREATE EXTENSION IF NOT EXISTS vector;`.
        *   Добавить поле `bio TEXT` в таблицу `users` (для исходного текста).
        *   Добавить поле `embedding vector(384)` в таблицу `users` (размерность `paraphrase-multilingual-MiniLM-L12-v2`).
        *   **Важно**: Добавить составной индекс для ускорения поиска по `university_id` + косинусному расстоянию (IVFFlat или HNSW, если записей много, но пока хватит сырых данных).

#### Этап 2: Обновление `bot.py` (Сбор данных)
*Цель: Научить бота собирать текстовое описание профиля ("О себе"), необходимое для генерации векторов.*

1.  **Модификация State Machine**:
    *   В `ConversationHandler` регистрации добавить шаг `REGISTER_BIO`.
    *   Попросить пользователя написать пару предложений о своих интересах.
2.  **Обновление DB-слоя (`db.py`)**:
    *   Обновить `add_or_update_user` и `update_user_profile`, чтобы они принимали и сохраняли `bio`.
    *   Добавить флаг `embedding_outdated = TRUE` (или просто проверять `NULL` в эмбеддинге), чтобы воркер знал, что нужно пересчитать вектор.

#### Этап 3: Создание ML Worker (Генерация эмбеддингов)
*Цель: Асинхронно превращать текст в векторы, не блокируя основной поток бота.*

1.  **Структура воркера**:
    *   Скрипт `worker.py`, работающий в бесконечном цикле (или по расписанию `schedule`).
2.  **Логика векторизации**:
    *   Загрузка модели `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`.
    *   SQL-запрос: `SELECT user_id, bio FROM users WHERE embedding IS NULL AND bio IS NOT NULL`.
    *   Генерация вектора.
    *   SQL-апдейт: `UPDATE users SET embedding = %s WHERE user_id = %s`.
    *   *Constraint*: Обработка батчами, чтобы не забить память.

#### Этап 4: Алгоритм Мэтчинга (Matching Job)
*Цель: Реализовать "умный" подбор пар на основе косинусного сходства.*

1.  **Логика выбора кандидатов**:
    *   Мы не можем мэтчить всех со всеми. Нужна механика "Активного поиска".
    *   Решение: Использовать существующую таблицу `coffee_requests`. Если статус `pending` и нет партнера — это кандидат на авто-мэтчинг.
    *   *Альтернатива*: Если пользователь просто "Active", это опасно (спам). Будем мэтчить только тех, кто создал заявку на **одинаковые интервалы времени** (или пересекающиеся) или тех, кто нажал кнопку "Хочу случайный кофе".
    *   **Принятое решение для v2.0**: Реализуем джоб, который смотрит на открытые `pending` заявки в **одном временном слоте** (например, +- 1 час) и **одной локации** (если важно) или просто ищет наиболее похожих людей, подавших заявки на "свободное время".
2.  **Жадный алгоритм**:
    *   Выгрузить активные заявки с векторами создателей.
    *   Посчитать матрицу расстояний (Cosine Similarity) через `sklearn.metrics.pairwise` или на стороне БД через оператор `<=>` (pgvector).
    *   *Фильтрация*:
        *   `university_id` должен совпадать (Strict).
        *   Исключить пары, которые уже встречались (проверка по истории встреч).
    *   Сформировать пары "Best Match".
3.  **Запись результатов**:
    *   Обновить `coffee_requests`: проставить `partner_user_id` и статус `matched`.

#### Этап 5: Интеграция и Уведомления
*Цель: Чтобы пользователи узнали, что ML нашел им пару.*

1.  **Polling изменений**:
    *   Бот должен узнавать, что заявка перешла в статус `matched` "извне" (воркером), а не через callback-кнопку.
    *   У вас уже есть `send_confirmations_job`. Нужно добавить (или адаптировать) джоб `notify_new_matches`, который ищет заявки, где `status='matched'`, но уведомление еще не отправлено (потребуется флаг `is_match_notification_sent` в БД, чтобы не спамить).

#### Этап 6: Чистка и Тесты
1.  Удаление старого кода ручного выбора (если решаем полностью уйти в ML) или оставление его как фолбэка.
2.  Тестирование на двух разных `university_id` (проверка изоляции).
